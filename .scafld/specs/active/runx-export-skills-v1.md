## Summary

Add `runx export <claude|codex> [skill-ref...] [--project] [--json]` to
register local runx skills as thin host-agent shims. The shims must delegate
execution back to `runx skill ...` so policy, approvals, and signed receipts
remain owned by runx.

Detailed design context is captured in
`../.plans/active/runx-export-skills.md` from the parent workbench. This spec is
the executable contract for the OSS code changes.

## Objectives

- Export all public project skills, or explicitly named skill refs, to Claude
  Code and Codex global skill directories.
- Support Claude project export to `.claude/skills/<name>/SKILL.md`.
- Fail closed for Codex project export until the project-level Codex path and
  rules behavior are verified live.
- Generate only interface shims, not copied runx skill implementations.
- Merge the Codex permission rule into `~/.codex/rules/default.rules` with a
  fenced idempotent block.
- Prune only files previously generated by this export command; never touch
  unmarked host skills or user-managed rules.

## Scope

- In scope:
  - Rust CLI command parsing/dispatch for `runx export`.
  - Runtime-owned skill export loading built on existing parser APIs and
    runtime patterns.
  - Target-specific file planning for Claude and Codex.
  - Shared idempotent write/prune helpers.
  - Shared CLI argument helpers where repeated parser code was already present.
  - Focused Rust tests for CLI parsing, file output, Codex rule merging, and
    prune safety.
- Out of scope:
  - Hosted/cloud export surfaces.
  - Codex project export enablement before live verification.
  - New skill manifest attributes.
  - New flags beyond `--project` and `--json`.
  - Copying or executing host-agent implementations directly.

## Dependencies

- Existing parser APIs, consumed through `runx-runtime` so `runx-cli` does not
  take a direct `runx-parser` dependency.
- Existing CLI launcher patterns in `crates/runx-cli/src/launcher.rs`.
- Existing runtime/CLI home-directory helper if present; otherwise add one
  shared helper rather than duplicating raw `HOME` logic.

## Assumptions

- The working directory is the skill project root, matching existing `runx
  skill` and `runx list skills` behavior.
- Global Claude skills live under `~/.claude/skills`.
- Global Codex skills live under `~/.codex/skills`, and Codex auto-loads
  `~/.codex/rules/default.rules`.
- Codex project scope remains unverified and therefore must return a clear
  error instead of writing files.
- Explicit positional refs override `catalog.visibility: internal`.

## Touchpoints

- `crates/runx-cli/src/main.rs` or the current native action dispatch module
- `crates/runx-cli/src/cli_args.rs` (new shared CLI argument helpers)
- `crates/runx-cli/src/export.rs` (new)

- `crates/runx-cli/src/mcp.rs` (remove duplicate argument helpers while preserving behavior)
- `crates/runx-cli/src/config.rs` and `crates/runx-cli/src/history.rs`
  (remove duplicate argument helpers while preserving behavior)
- `crates/runx-runtime/src/export.rs` (new runtime-owned skill export loader)
- `crates/runx-runtime/src/lib.rs`
- `scripts/generate-cli-feature-parity.ts` and generated CLI parity fixtures
- `crates/runx-cli/tests/export.rs` (new)
- `crates/runx-cli/Cargo.toml` if a direct internal dependency is required

## Risks

- Accidentally installing full skill instructions into host agents would bypass
  runx governance. Shims must clearly instruct delegation only.
- Host skill directories may contain hand-authored files. Prune logic must key
  only off runx export markers.
- Codex rules files may contain user approvals. The managed block must be
  replaced in place without mutating surrounding content.
- The repo is currently dirty with unrelated runx work; edits must stay inside
  the declared touchpoints unless a direct compile/test dependency requires a
  narrow additional file.
