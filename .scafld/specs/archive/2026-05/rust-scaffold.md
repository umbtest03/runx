## Summary

Port the workspace scaffold (`runx init`) and skill scaffold (`runx new`)
to Rust. Today these live in `packages/cli/src/commands/init.ts`,
`packages/cli/src/commands/new.ts`, and `packages/cli/src/scaffold.ts`,
backed by `packages/create-skill/`.

This is a CLI cutover task, not a scaffold redesign. The Rust implementation
must preserve the existing `runx init` / `runx new` command surface, JSON
result shape, generated bytes, and non-overwrite behavior so later CLI sunset
work can route these commands natively without changing user-visible
scaffolds.

## Implementation Notes

- Keep deterministic scaffold rendering pure in `runx-runtime`; CLI modules
  parse args, resolve roots from `RUNX_CWD` / `INIT_CWD` / cwd, call the
  runtime, and render JSON/human output.
- Preserve TS path resolution:
  - explicit `--directory` is absolute as-is, otherwise resolved against
    `RUNX_CWD ?? INIT_CWD ?? cwd`;
  - default `runx new` directory is the sanitized package name under the same
    root;
  - project init uses the existing runx project directory resolution.
- Preserve TS sanitization:
  `trim().toLowerCase().replace(/[^a-z0-9_.-]+/g, "-").replace(/^[._-]+|[._-]+$/g, "") || "runx-package"`.
- Preserve packet namespace derivation: lower-case, strip leading `@`, replace
  non-alphanumeric runs with `.`, trim `.`, default `runx.package`.
- Use typed Rust structs for scaffold results and fixture oracles; do not test
  production behavior through untyped `serde_json::Value` blobs except at the
  final CLI boundary where JSON text is parsed.
- The fixture generator should execute/read the current TS implementation and
  write canonical fixture expectations. Rust tests consume checked-in fixture
  oracles; they must not regenerate expectations during test runs.
- If live timestamp/UUID generation is needed for init state, tests must inject
  deterministic clock/ID providers. Production defaults must use real
  time/IDs.
